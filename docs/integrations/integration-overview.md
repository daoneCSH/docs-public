# Integration Overview

DADP는 세 가지 공식 연동 경로를 제공한다. 이 세 경로는 같은 기능을 중복 구현한 것이 아니라, 서로 다른 실행 경계에 맞게 같은 런타임 실행면을 연결하는 구조다.

## 공식 연동 경로

| 경로 | 개입 위치 | 대표 사용 상황 |
|------|-----------|----------------|
| Wrapper | 애플리케이션 JDBC 경계 | SQL 경로를 투명하게 보호 |
| Direct API | 애플리케이션 서비스 코드 | 코드에서 명시적으로 암복호화 호출 |
| DB UDF | 데이터베이스 SQL 경계 | DB 내부 절차와 SQL 배치 보호 |

## 공통 처리 모델

모든 공식 연동 경로는 결국 같은 실행면을 공유한다.

1. 연동 경로가 실행 시점을 결정한다.
2. 필요한 정책 또는 매핑 정보를 참조한다.
3. Engine에 실행 요청을 보낸다.
4. 결과를 원래 실행 경계로 되돌린다.

## 선택 기준

### Wrapper를 고르는 경우

- 애플리케이션 코드 수정량을 줄이고 싶을 때
- JDBC 경계에서 보호를 적용하고 싶을 때

### Direct API를 고르는 경우

- 암복호화 시점을 코드에서 명시적으로 제어하고 싶을 때
- 동일 애플리케이션 안에서 데이터 범위별로 호출 제어가 필요할 때

### DB UDF를 고르는 경우

- 데이터베이스 내부 함수나 배치 SQL에서 직접 보호가 필요할 때
- 보호 로직을 DB 계층에 두고 싶을 때

## 운영 관점의 차이

- Wrapper는 인스턴스 등록과 스키마 동기화가 중요하다.
- Direct API는 호출 시점과 요청 본문 계약이 중요하다.
- DB UDF는 설치 오브젝트와 네트워크 허용 상태가 중요하다.

따라서 세 연동 경로는 같은 “암복호화 기능”을 제공하더라도 운영자가 점검해야 하는 대상이 다르다.
